# FINDINGS.md — PocketMonstersStadiumRecomp analysis log

Durable, append-only record of reverse-engineering conclusions for PMS-J
bring-up. Each entry states **what** we concluded, **why** (the evidence),
and a **confidence** label so future work knows how much to trust it.

This is the project-level home for analysis notes. Companion locations:
- **Inline `game.toml` comments** — the rationale for each `[patches]` entry
  lives next to the patch itself (the authoritative spot for build decisions).
- **Ghidra DB** — function renames + plate comments. When the analyzed PMS
  Ghidra project is loaded, mirror the identifications below onto the symbols
  (see each entry's "Ghidra labels to apply").
- `tools/` converters and `lib/` upstreams — code-level fixes.

## Confidence convention

- **HIGH** — multiple independent lines of evidence converge (byte/size match
  to a known reference, address adjacency, algorithm identity). Safe to build on.
- **MEDIUM** — strong single line of evidence or close-but-not-exact match.
  Usable, but re-verify before a load-bearing decision.
- **LOW** — hypothesis / pattern guess. Record it, but prove it before acting.

The primary reference oracle is **PokemonStadiumRecomp (PSR)** — Pokemon
Stadium (U), which has a full pret C decompilation in `../PokemonStadiumRecomp/
disasm/`. PMS-J and PSR-US are the same engine (near-identical boot layout:
`Idle_ThreadEntry` at the *same* address `0x80000460`; `Main` at
`0x80000530`/PSR vs `0x80000550`/PMS), so PSR's decompiled source is a strong
basis for pattern-matching unnamed PMS functions.

## CP-001 — `FUN_800095cc` / `FUN_80009490` are libleo 64DD bootdisk, NOT copy protection. Confidence: **HIGH**

**What.** The stubbed `FUN_800095cc` is libleo's `__LeoBootGame2` (a 64DD disk
bootstrap that is scrambled in ROM). `FUN_80009490`, immediately before it, is
`LeoBootGame` — the descramble routine that decrypts `__LeoBootGame2` in place
and then jumps to it. This is **64DD boot residue**, not the game's copy
protection. It is **not on the cart-boot path** and stubbing it is safe.

**Why (evidence vs `PSR/disasm/src/libleo/bootdisk.c`, the descrambled source):**
- `FUN_80009490` size = **0x13C**, exactly matching `bzero(LeoBootGame, 0x13C)`
  in bootdisk.c.
- `FUN_800095cc` sits at `0x80009490 + 0x13C` — precisely the function
  `LeoBootGame` descrambles (`ptr = (u8*)&__LeoBootGame2`).
- The descramble algorithm is byte-identical: `key = (sum of the 4 address
  bytes) & 0xFF`, then `ptr[2] -= key; ptr[3] += key` over 0x324 bytes for
  `__LeoBootGame2`, then 0x60 for `__LeoBootGame3`.
- PSR keeps these same bytes scrambled in ROM (Makefile: *"this file is
  encrypted we cant have splat decrypt it"*, force-links `bootstrap.s.o`).
- Historical context (Matthew): PMS was a 64DD-era title (alleged Disk Drive
  addon), which explains why a *live* libleo bootdisk path exists at all.
- `LeoBootGame` is called only from `func_800296AC`, reached only via the
  `STATE_N64DD_BOOT_UNUSED` case of the Game_Thread state machine — a state
  nothing sets on a normal cart boot.

**Ghidra labels to apply (analyzed PMS DB):**
- `0x80009490` → `LeoBootGame`  (plate comment: "libleo 64DD bootdisk
  descramble; confidence HIGH — size 0x13C + algo match to PSR bootdisk.c")
- `0x800095cc` → `__LeoBootGame2`  (plate comment: "scrambled 64DD bootstrap;
  stubbed in game.toml; never runs on cart boot")
- `0x800095cc + ...` → `__LeoBootGame3` (0x60, descrambled by LeoBootGame's
  second loop)

Not added to `function_renames.txt` because that file is auto-generated by
`tools/match_libultra.py` (manual lines would be clobbered) and the stub keys
off the current `FUN_800095cc` symbol; revisit if/when the rename pipeline gains
a manual-overrides file.

## CP-002 — Real copy protection is `Game_DoCopyProtection`; it PASSES on PMS and cannot cause a black screen. Confidence: **HIGH**

**What.** PSR's actual copy-protection check is `Game_DoCopyProtection`
(PSR `0x80028FA0`), called at the top of the Game_Thread state-machine loop:

```c
if (!(PI_STATUS_REG & 3) && (*(u32*)0xB0000E38 & 0xFFFF) != 0x828A) state = -0x10;
```

On PMS this check **passes**, and even if it failed it could not produce a
black screen.

**Why:**
- PMS-J ROM bytes at offset `0xE38` are **byte-identical** to PSR-US
  (`...1060 828a...` → `*(u32*)0xE38 & 0xFFFF == 0x828A`). The magic matches.
- Decisive regardless of the check: the state-machine `switch` `default:` case
  runs `func_80029310()` — the **same N64-logo intro** as the normal path. A
  failed copy-protection check therefore loops the N64 logo, never blanks.

**Implication.** PMS's black screen is upstream of the state machine — in the
idle-thread init chain or `Game_Thread` startup, before the first graphics
`send_dl`. Copy protection is ruled out. (Diagnosis in progress via the
always-on libultra-call ring — see DEBUG notes / watchdog in
`rt64_render_context.cpp`.)

## BOOT-001 — Black screen root cause: preemption checkpoints gated on trace_mode (2026-06-09)

**Conclusion (HIGH):** The boot deadlock was the cooperative scheduler starving:
every guest thread blocked on a queue while the idle thread spun in call-free
code; externally-posted messages (VI retrace from the host VI thread) were only
drained at guest os-calls or at `ultramodern_scheduler_tick()` checkpoints — and
the recompiler emitted those checkpoints (`TRACE_LOOP()`) on loop back-edges
**only when `trace_mode = true`**. PMS had `trace_mode = false`.

**Evidence:** always-on ultra-trace ring: 79 os calls in first 2ms then total
silence; `~vi_retrace` deliveries "succeed" every frame (external enqueue
returns 0 unconditionally) while the lone `osRecvMesg(0x800AA100)` waiter never
logs a `.done`. PSR boots with `trace_mode = true` (same runtime).

**Fix (engine, correct-by-construction):** N64Recomp now emits
`RECOMP_LOOP_CHECKPOINT()` on every loop back-edge unconditionally; the default
definition lives in `N64Recomp/include/recomp.h` and routes to
`ultramodern_scheduler_tick()`. `trace_mode` gates only TRACE_ENTRY/TRACE_RETURN
logging.

## BOOT-002 — Truncated function sizes silently dropped final delay slots (2026-06-09)

**Conclusion (HIGH):** 70+ functions in `functions.json`/symbols had sizes that
ended exactly at a final branch/jump, cutting off its delay slot. N64Recomp's
`process_delay_slot` silently skipped the out-of-range delay slot, vanishing the
instruction's effect. Flagship case: the game's Yay0 decompressor
`FUN_80048010` (size 0xA8) lost `addi t3,t3,0x12` @ 0x800480B8 — extended-length
backrefs lost the +0x12 bias, a length byte of 0 wrapped the copy counter to
~4G, and the decompressor scribbled rdram for ~3.5s until it clobbered the
scheduler's OSMesgQueue at 0x800AA100 (caught by do_send's corruption guard).
Oracle: `tools/test_yay0_oracle.py` decompresses the same ROM blob
(0xB98090, 0x2FC0 in) to 0x3264 bytes in 10379 ops — input data was fine.

**Fixes:** (1) engine: `process_delay_slot` now hard-errors when the delay slot
is out of bounds, and the previously-unchecked `print_return_with_delay_slot()`
/ `print_func_call_by_register()` results are propagated (jr-ra delay slots
were silently dropped too). (2) data: `tools/fix_truncated_delay_slots.py`
scans every function's last word in the ROM and extends sizes (+4) — appended
to `function_sizes.txt`; fixpoint verified (re-run reports none).

## BOOT-003 — Current frontier: resource-server request/reply layer (2026-06-09)

Boot now completes the entire resident init: scheduler thread (0x80004D14;
client msgs SP=0x64 DP=0x65 retrace=0x66 PRENMI=0x68 on queue 0x800AA100) alive
on retrace; libleo concludes no-drive via HLE osEPiReadIo; audio engine init
completes (heap @0x800B41F8, Yay0 module decompression, alInit via
FUN_800371a4 with HLE osAiSetFrequency); osContInit + osEepromProbe + 26 PI
DMAs; main thread (FUN_8000a92c, scheduler client queue 0x800AAE48, "VTRE"
retrace messages) runs ≥1 main-loop iteration and builds the first gfx task
(FUN_8003b868).

**Stall (MEDIUM understanding):** the pri-10 resource-server thread
(FUN_80000a58; cmd queue 0x8008701C; shared reply queue 0x800A96F0) serves
chunked 0x1000-byte ROM reads (FUN_80003824 → FUN_80000d40 post + blocking
recv) and resource decode (FUN_80003a24; magics "PERS-SZP" → SZP decompress
FUN_80003978, "PRESJPEG" → JPEG FUN_800038e0). A second-round request blocks
forever on 0x800A96F0. Thread 7 (entry 0x80017700, queue 0x800AD170, created
lazily by FUN_800177dc) never receives anything. Per-thread attribution is the
blocker: **the rings need a thread-id field** (next instrumentation step).
After that: overlay/fragment sectioning (num_sections=1 today) is the known
next phase before graphics can flow.
